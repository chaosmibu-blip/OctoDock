## 二、程式碼修正（可立即修）

### P1：Notion append_blocks formatResponse 遺漏
- **來源**：🧪 全功能測試報告 + GitHub 測試報告補充觀察
- **問題**：`formatResponse` 沒有 `append_blocks` 的 case，掉進 default 回傳 raw JSON
- **修法**：在 `notion.ts` formatResponse 的寫入類 case 組裡加上 `case "append_blocks":`
- **影響**：1 行
- **檔案**：`src/adapters/notion.ts:885`

### P2：Notion help 計數不正確
- **來源**：🧪 全功能測試報告
- **問題**：getSkill() 概覽寫「19 actions」，但 actionMap 實際有 21 個（append_content + move_page 有在裡面，但概覽數字沒更新）
- **修法**：更新 getSkill() 裡的 action 計數為 21
- **影響**：1 行
- **檔案**：`src/adapters/notion.ts` getSkill 函式

### P3：排程引擎 DB 問題
- **來源**：🧪 全功能測試報告
- **問題**：schedule_create 和 schedule_list 都因 DB insert/select 失敗
- **現況**：檢查兩個 DB 的 schedules 表都存在且 schema 正確（13 個欄位）
- **修法**：需要跑一次 schedule_create 的 debug 來找出實際錯誤。可能是 `id` 欄位的 default UUID 沒生效，或 Drizzle ORM 的 query 格式有問題
- **影響**：需要 debug
- **檔案**：`src/services/scheduler.ts` + `src/mcp/system-actions.ts`

### P4：Notion query_database 不顯示欄位值
- **來源**：🧪 全功能測試報告
- **問題**：query_database 的 formatResponse 只回傳標題和 ID，不顯示各欄位的值
- **修法**：在 `summarizeSearchResults` 或 query_database 的 formatResponse 裡加上欄位值的提取
- **影響**：中等（需要解析 Notion 的 property value 格式）
- **檔案**：`src/adapters/notion.ts` formatResponse query_database case

### P5：Notion get_comments 失敗
- **來源**：🔧 Notion 測試報告、🧪 全功能測試報告
- **問題**：add_comment 成功但 get_comments 報錯「Could not find block」
- **可能原因**：endpoint 用了 block_id 而不是 page_id，或 Notion integration 權限未含 read_comments
- **修法**：檢查 get_comments 的 API endpoint 是否正確（應該是 `/v1/comments?block_id=xxx`）
- **檔案**：`src/adapters/notion.ts` execute notion_get_comments case

## 三、架構層面（已實作但可優化）

### A1：記憶 Onboarding 觸發邏輯強化
- **來源**：⚠️ AI 對用戶的個人記憶查詢功能
- **現況**：`import_memory` action 已實作，help 裡的 onboarding 提示已有
- **缺口**：
  - 觸發條件不夠完整：目前只在「沒有記憶」時觸發 onboarding，但應該在「記憶超過 30 天沒更新」或「連接新 App」時也觸發
  - suggestion 格式可以更好（目前是一大段，可以更簡潔）
- **優先級**：低（現有功能堪用）

### A2：混合模式 http_request 權限控制
- **來源**：⚠️ B2B、開發者生態、API 市集
- **現況**：`http_request` system action 已實作，有基本安全檢查（只允許 HTTPS、阻擋內網）
- **缺口**：沒有自動帶用戶的 App token（目前是裸 HTTP 請求，不帶 OAuth token）
- **優先級**：低（目前作為補充工具夠用）
