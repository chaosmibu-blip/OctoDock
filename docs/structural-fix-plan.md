# 結構性修復規劃：讓系統不可能靜默地壞掉

> 日期：2026-03-25
> 狀態：**規劃中（等待確認後執行）**
> 來源：Production DB 資料完整性稽核發現的系統性問題

---

## 發生了什麼事

我們在準備執行一個較大的產品演化計畫時，回頭檢查了 production 資料庫，發現大量欄位是空的。這些欄位在程式碼裡定義了，但從來沒有被實際寫入資料。有些空了幾個月都沒人發現。

具體來說：
- 記憶系統的語意搜尋功能（embedding 欄位）：143 筆記憶裡面，0 筆有向量嵌入。這代表語意搜尋這個核心功能從上線以來就是壞的，一直退化到文字比對在運作。
- 用戶在各 App 裡的身份（app_user_id / app_user_name）：51 筆連結紀錄，0 筆有值。代表系統不知道用戶在 Gmail 裡是哪個 email、在 GitHub 裡是哪個帳號。
- 操作意圖（intent）：1520 筆操作裡只有 97 筆有記錄意圖（6.4%）。代表系統不知道用戶為什麼做這些操作，無法分析行為模式。
- 用戶詢問的困難點（difficulty）：只有 9 筆有值（0.6%）。用戶問了問題但系統沒有記下來。

## 為什麼會這樣

根本原因不是某一個功能寫錯了，而是整個系統缺乏三個能力：

**一、沒有辦法確保「每條路都有記錄」。** 系統處理一個請求有很多條路——成功的路、失敗的路、提前返回的路、查詢幫助的路。每條路都應該記錄下來，但現在的程式碼是在每條路的末端各自決定要不要記錄。結果就是有些路記了，有些路忘了。忘了的那些操作就像從來沒發生過，完全看不見。

**二、沒有辦法確保「定義的欄位真的有程式碼在寫入」。** 資料庫的表格設計了很多欄位，但設計歸設計，有沒有程式碼真的把資料寫進這些欄位，沒有人檢查。就像蓋了一棟大樓，圖紙上畫了水管，但實際施工時有些水管根本沒接上，直到住戶打開水龍頭才發現沒水。

**三、沒有辦法自動發現「什麼時候壞了」。** 系統壞了不會主動告訴任何人。要靠人手動去查資料庫才知道。如果沒有人去查，問題就會一直累積，直到在最關鍵的時候爆發——就像這次，我們要執行一個大計畫才發現地基有問題。

## 三個結構性解法

這三個解法不是修某個 bug，是讓上面三種問題在結構上不可能再發生。

---

### 解法一：單一出口模式 — 讓「忘記記錄」不可能發生

#### 為什麼現在會忘記

想像一個辦公室：每個員工處理完案件後，要自己走到檔案室去歸檔。有些員工記得，有些忘了。忘了的案件就消失了，沒有人知道它曾經被處理過。

現在的程式碼就是這樣。處理請求的函式有 23 個出口（12 個在 do，11 個在 help），每個出口要自己記得呼叫記錄函式。有些記得帶齊所有資訊，有些少帶了一兩個欄位，有些根本沒呼叫。

#### 怎麼解決

改成「不管走哪條路，最後都經過同一個出口，出口負責歸檔」。就像辦公室改規則：員工不用自己去歸檔，所有案件處理完放在桌上，統一由一個歸檔員收走。這樣不可能有案件漏掉。

具體來說：
1. 所有路徑不再直接回傳結果，而是把結果存到一個變數裡
2. 最後統一經過一個出口函式
3. 出口函式負責把所有可用的資訊（誰做的、做了什麼、結果是什麼、花了多久）一次性寫進資料庫
4. 個別路徑不再各自記錄，避免有的記有的不記

#### 有一個要注意的地方

目前「真正執行 API 呼叫」的那條路（executeWithMiddleware）已經有自己的記錄邏輯，而且它記錄的資訊最完整（包含實際的 API 回傳結果）。出口函式不能重複記錄這筆。解法是用一個旗標標記「這筆已經被記錄過了」，出口函式看到旗標就跳過。

#### 資料流

每次請求進來時就有的資訊：
- 是哪個用戶（userId）
- 要操作哪個 App（app）
- 要做什麼動作（action）
- 帶了什麼參數（params）
- 為什麼要做（intent，octodock_do 時）
- 卡在哪裡（difficulty，octodock_help 時）
- 是哪個 AI 在操作（agentInstanceId）
- 開始的時間（startTime）

每條路徑執行後產生的資訊：
- 執行結果（result）
- 成功還是失敗（success）
- 花了多少時間（durationMs）

以上所有資訊都要寫進 operations 表。不管是成功、失敗、提前返回、查詢幫助，一筆都不能漏。

#### 受影響的檔案

| 檔案 | 改動 |
|------|------|
| server.ts | 所有 return 改為賦值，最後統一出口 |
| logger.ts | executeWithMiddleware 設定「已記錄」旗標 |

#### 怎麼驗證做對了

改完後去 production 資料庫查：
- 呼叫一次 octodock_help → 確認 operations 表有這筆記錄
- 故意觸發一個不存在的 action → 確認失敗記錄裡有 agentInstanceId
- 做一次 dry-run → 確認有記錄

---

### 解法二：Schema-Code 契約檢查 — 讓「定義了但沒寫入」被自動攔住

#### 為什麼現在會漏

資料庫的表格設計（schema）和實際寫入資料的程式碼是分開維護的。加一個新欄位只需要改 schema，但讓資料真的寫進去需要改好幾個地方的程式碼。如果只改了 schema 沒改程式碼，或者改了部分程式碼沒改全部，這個欄位就會一直是空的。

而且沒有任何機制會告訴你「這個欄位從來沒有被寫入過」。就像圖書館新增了一個書架，但沒有人把書放上去，也沒有人檢查書架上有沒有書。

#### 怎麼解決

建立兩層自動檢查：

**第一層：靜態檢查。** 自動掃描 schema 裡的每個欄位，然後在程式碼裡搜尋有沒有對應的寫入操作。如果某個欄位在 schema 裡定義了，但程式碼裡找不到任何寫入它的地方，就報警。這在 build 之後自動執行。

**第二層：動態檢查。** 連到實際的資料庫，查每個欄位的填充率（有多少比例的記錄裡這個欄位有值）。如果一個欄位的填充率低於預期（比如 intent 應該要 > 80%，但實際只有 6.4%），就報警。

#### 受影響的檔案

| 檔案 | 改動 |
|------|------|
| 新增 `scripts/check-schema-coverage.ts` | 靜態檢查 |
| 新增 `scripts/check-data-integrity.ts` | 動態檢查 |
| package.json | 新增執行指令 |

---

### 解法三：Health Check — 讓「壞了但不知道」被自動發現

#### 為什麼現在不知道壞了

系統沒有自我檢查的能力。就像一台車沒有儀表板——油快沒了不知道，引擎過熱不知道，輪胎沒氣不知道。要等到拋錨了才發現。

今天我們手動查了資料庫才發現大量問題。如果沒有這次手動檢查，這些問題會繼續靜默地存在。

#### 怎麼解決

建一個健康檢查端點，就像車子的儀表板。隨時可以看系統的健康狀態：

- 每個表有多少筆資料
- 每個關鍵欄位的填充率是多少
- 有沒有異常（填充率低於預期的標紅）
- 最後一次操作是什麼時候（判斷系統是否還在正常運作）

這個端點需要管理員權限才能存取，不能公開（因為包含系統內部的統計資訊）。

未來可以讓它定期自動執行，異常時主動通知，而不是等人去看。

#### 受影響的檔案

| 檔案 | 改動 |
|------|------|
| 新增 `src/app/api/health/data/route.ts` | Health Check API |

## 除了結構性修復，還有現存的壞資料需要修

上面三個解法是「防止未來再發生」的結構。但已經壞掉的資料也需要處理：

**記憶系統的語意搜尋（embedding 全空）：** 需要查清楚為什麼儲存記憶的函式沒有產生向量嵌入。修好後，把現有的 143 筆記憶補上向量。這樣語意搜尋才能真正運作。

**用戶在各 App 的身份（app_user_id 全空）：** 需要在用戶連結 App 的流程中加入「取得用戶在該 App 裡的身份」這一步。現有的 51 筆連結，需要用已存的 token 重新查詢一次補上。

**操作意圖（intent 6.4%）：** 早期的歷史資料無法回補（當時還沒有 intent 欄位）。但解法一上線後，所有新的操作都會有 intent 記錄。

**登入帳號的 refresh token（全空）：** 需要確認 Google 登入的設定有沒有要求離線存取權限。如果沒有，用戶的登入 token 過期後就需要重新登入。

---

## 執行順序

1. 解法一（單一出口）— 最優先，因為這是讓資料流不再斷裂的基礎。沒有這個，後面修的東西還是可能再壞
2. 修復 embedding + app_user_id — 把現有的壞資料補好
3. 解法二（契約檢查）— 防止未來再出現「定義了欄位但沒程式碼寫入」的問題
4. 解法三（health check）— 讓所有異常自動浮現，不再靠人記得去查
