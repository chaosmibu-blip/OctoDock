# OctoDock 產品演化規劃：短期 → 中期 → 長期

> 日期：2026-03-25（初版）→ 2026-03-26（根據討論結論重寫）
> 來源：協作有局產品思考討論 + 架構分析 + Production 數據稽核
> 狀態：**設計審查中（等待聖堯確認）**

---

## 這份文件在說什麼

OctoDock 現在是「AI 的工具連接層」——讓 AI 透過一個 URL 操作用戶已授權的所有 App。這份文件規劃的是接下來三個階段要怎麼演化，從工具連接層變成記憶與上下文層，最後變成 AI 時代的服務基礎設施。

演化路徑：

- **短期（現在）**：工具連接層 — 連接既有 App 的 API
- **中期**：記憶與上下文層 — 事件圖譜 + 上下文接續 + SOP 強化 + 企業內部管理
- **長期**：服務基礎設施 — 服務 API + 知識技能市場 + 信任層 + 供需洞察

五個模組：

- **A. 事件圖譜**（中期）— 記錄 AI 代表用戶做了什麼，讓用戶看得見
- **B. 上下文接續**（中期）— AI 換了 session 時，自動補上之前做過什麼
- **C. 服務 API**（長期）— 讓用戶和商家也能上傳自己的服務，讓別人的 AI 找到並呼叫
- **D. 知識技能市場**（長期）— 把用戶的專業認知打包成可出租的模組
- **E. 信任層 + 供需洞察**（長期）— 從操作數據累積服務品質，成為 AI 時代的服務 DNS

---

## 我們已經有什麼

OctoDock 現有的基礎可以直接延伸：

- 每次操作都記錄在 operations 表裡（誰做的、用哪個 App、做了什麼、結果如何、花了多久）
- 記憶系統已經能存用戶偏好、操作模式、名稱對應
- SOP 偵測器已經能從重複操作中自動發現工作流程
- Adapter 機制讓新 App 可以即插即用
- middleware pipeline 可以在操作前後注入邏輯（查現狀、建議下一步、修正參數）

缺少的：操作之間的因果關係、用戶看得見的操作歷史、AI 換手時的上下文補充、用戶自行上傳服務的機制、知識打包和信任追蹤。

## 模組 A：事件圖譜

### 要解決什麼

用戶不知道 AI 代表自己做了什麼。現在的操作紀錄只存在資料庫裡，沒有介面讓用戶看到。而且操作紀錄是一筆一筆散落的，看不出「先搜尋了 email，然後把內容寫進 Notion」這種前後因果。

### 怎麼做

不需要建新表。現有的 operations 表已經記錄了每次操作的完整資訊，加上時間戳（createdAt）就能按時間分組。

分組方式很簡單：同一個用戶的操作，如果兩筆之間間隔超過 30 分鐘，就算不同的工作階段。這跟現有的 SOP 偵測器用的邏輯一樣。不需要額外存「這筆操作屬於哪個 session」，因為從時間就能算出來。

因果關係也不需要靠 AI 猜，用確定性規則就好：如果後一筆操作的參數裡包含前一筆操作結果中的 ID，那它們就是因果關係。例如先搜尋得到一個頁面 ID，然後用這個 ID 去讀取頁面，這就是因果。

需要加一個欄位 `parentOperationId`，記錄「這個操作是因為哪個操作觸發的」。

### 用戶看到什麼

在 Dashboard 新增「操作歷史」頁面。按工作階段分組，每個階段顯示摘要（例如「下午 2:00-2:30，操作了 Notion 和 Gmail，共 8 個動作」）。展開可以看每個動作的 App 圖示、動作名稱、一句話結果描述、時間。因果關係用縮排或連線表示。不顯示技術細節（raw JSON），只顯示人看得懂的內容。

## 模組 B：上下文接續

### 要解決什麼

用戶換了 AI session（關掉 Claude 再開，或從 Claude 換到 Cursor），新的 AI 不知道之前做了什麼。同樣的工作流程，AI 之前都做對，突然做錯了，很可能是因為換了 session 失去了上下文。

### 怎麼偵測

不靠 header 或時間間隔判斷「是不是換了 AI」，因為目前 MCP 協議沒有可靠的 client identity（production 數據顯示所有 Claude.ai 請求都帶同一個 ID，無法區分）。

改用**行為異常**偵測：如果 AI 一直都用正確的方式呼叫某個操作，突然用了不同的方式或傳了不同格式的參數，大概率是換了 session 失去了上下文。這時候 OctoDock 主動補上「你之前都是這樣做的」。

三種異常模式：
- 呼叫了一個從沒用過的 action，但有一個很像的常用 action
- 參數格式突然不同（之前都傳物件，這次傳字串）
- 重複執行已完成的操作（30 分鐘前剛建過同名頁面）

### 怎麼補上下文

偵測到異常時，在回應裡附加精準的 3-5 行提示。例如：「你之前都用 get_events 這個 action（用了 47 次），這次用了 list_events，這是同一個操作的別名，正常執行。」或者：「你 2 小時前用 Claude 在 Notion 建了 3 個頁面，這是新的工作階段。」

上下文放在回應的 context 欄位裡，不影響正常的操作結果。

## 模組 C：服務 API（輸出 API）

### 要解決什麼

現在 OctoDock 是單向的：用戶透過 API 取得別人的服務（Notion、Gmail、GitHub）。但反過來，用戶自己也能提供服務——記帳士提供稅務諮詢、餐廳提供外送、自由工作者提供設計。

本質是把「人」類比成「App」。一個 App 透過 API 讓 AI 知道它能做什麼，一個人（商家）也可以透過 API 讓 AI 知道他能提供什麼服務。商家不需要開店、不需要網站、不需要上架平台，只需要描述自己的服務，OctoDock 幫他產生標準化的 API，任何 AI 都能發現和呼叫。

### 一句話開店

商家不會寫 API。OctoDock 要做的是讓商家用自然語言描述服務（「我賣滷肉飯，一碗 50 元，營業時間 11-14 點，外送範圍宜蘭市區」），OctoDock 轉成結構化的服務描述（商品、價格、營業時間、服務範圍），任何 AI 看到就知道什麼時候該呼叫、怎麼呼叫。而且服務內部可以連回商家的 AI 做即時客服。

OctoDock 掌握的是格式標準——統一或客製化呼叫與回傳的內容格式，這本身就是收費項目。

### 三層權限（記帳士事務所案例）

同一套 API，三種角色三種權限：

**老闆**看到所有客戶資料、員工績效、財務數據，可以管理權限和對外開放的內容。

**員工**只看到自己負責的客戶，可以查進度、新增待辦、查帳務紀錄，不能看其他員工的資料。

**外部消費者的 AI** 只看到「提供什麼服務、價格多少、現在能不能接案」，可以發起諮詢，由事務所的 AI 即時回應，不能看到任何內部資料。

一個 API，三層權限，三種角色，全部透過 OctoDock 管理。OctoDock 同時是事務所的後台管理平台（對內）、事務所的對外通路（對外）、消費者 AI 找到服務的入口（市場）。

### 服務發現

外部 AI 怎麼找到 OctoDock 上的服務？三個管道：透過 octodock_help 擴充（回傳可用的外部服務）、公開的服務目錄 API（支援分類和地區搜尋）、服務目錄網頁（SEO 友善，Google 也能索引）。

### 跟現有架構的接合

服務 API 不是另起爐灶，是延伸現有的 adapter 機制。現有的 AppAdapter 介面、registry 自動掃描、action resolution、logOperation、param-guard 全部可以複用。新增一個簡化版的 ServiceAdapter 介面（不需要 OAuth），從 DB 動態載入。

## 模組 D：知識技能市場

### 要解決什麼

模組 C 讓用戶對外提供「服務」（需要人或人的 AI 即時執行）。模組 D 更進一步：讓用戶對外提供「認知」（打包好的 know-how，別人的 AI 呼叫後自動套用，不需要原始擁有者在線）。

類比：Fiverr 賣的是人的時間（有上限），知識技能市場賣的是人的認知模型（可無限複製）。

### 兩種場景

**非同步：** 一個旅行社老闆透過 OctoDock 累積的行程規劃 know-how（景點搭配、交通安排、預算分配），被打包成「旅行規劃 Skill」。其他用戶的 AI 規劃旅行時呼叫這個 Skill，得到專業水準的建議。老闆不需要在線。

**同步：** 一個中小企業主的 AI 跟帶有「稅務規劃 Skill」的 AI 對話，得到針對他公司狀況的節稅建議。

### 知識從哪來

不靠用戶手動打包。從 OctoDock 現有的 SOP、記憶、操作模式自動萃取。用戶審查編輯後發布。萃取過程去識別化——移除所有個人資訊，只保留結構化的判斷邏輯。

## 模組 E：信任層 + 供需洞察

### 要解決什麼

當 OctoDock 同時擁有三樣東西，它就不只是工具了：誰能做什麼（供給）、誰需要什麼（需求）、做得好不好（品質）。同時擁有這三樣的角色，在傳統世界裡叫做「市場」。

### 賣信任

每次服務被呼叫，事件圖譜記錄：AI 呼叫了什麼、實際做了什麼、用戶最終有沒有得到應有的服務。品質分數從行為數據自動累積——用戶沒有 undo 代表滿意、重複購買代表認可、立即修正代表不符預期。不靠商家自評或用戶主動打分。

### 賣洞察

OctoDock 知道每天有多少人在找某類服務、有多少人提供某類服務、各類服務的品質分布。匿名化後的供需趨勢可以給商家看（你的排名、需求變化），也可以給想了解市場的人看。

### 跟現有平台的差別

蝦皮/Uber/Airbnb 擁有流量，商家依賴平台。OctoDock 不擁有流量——流量在每個用戶的 AI 裡。OctoDock 擁有的是讓 AI 找到服務的能力和信任數據。更像 DNS 之於網際網路：DNS 不擁有任何網站，但沒有 DNS 你就找不到任何網站。OctoDock 是 AI 時代的服務 DNS。

## 五個模組怎麼互相連動

A（事件圖譜）是所有模組的數據基礎。沒有操作歷史，上下文接續就沒有資料可以補，信任層就沒有履約紀錄可以追蹤。

B（上下文接續）依賴 A 的數據來判斷行為異常和補充歷史上下文。對 C 的消費者 AI 也有用——第一次用某服務時，可以補充過去的偏好。

C（服務 API）是商業模式的翻轉——從消費者變成服務提供者。D（知識市場）是 C 的終極形態——從提供服務變成提供認知。

E（信任層）依賴 A 的數據和 C/D 的服務，反過來用品質分數和供需洞察回饋給商家，形成正循環。

## 待確認的問題

1. 事件圖譜的操作歷史頁面要多細？只看工作階段摘要，還是能展開看每一步？
2. 服務 API 的 contactEndpoint：商家的 AI 客服由 OctoDock 代管，還是商家自己提供？
3. 知識技能的定價模型：按次、月費、還是抽成？誰定價？
4. 供需洞察的開放程度：匿名趨勢數據免費公開還是付費？
5. 金流結算：AI 代替用戶付款用現有金流還是需要新機制？
6. 冷啟動策略：服務目錄一開始沒有商家怎麼辦？
