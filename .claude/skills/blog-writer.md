# OctoDock Blog 文章產生器

三種文章類型 + Claude 專欄。每篇從使用者視角出發，每批產完後自我審查、回饋寫到 Notion。

---

## 含金量標準

讀者看完一篇，要有至少一個他自己想不到的東西。

**有含金量：** 可直接複製的 prompt、操作順序的設計邏輯（為什麼先 A 再 B）、真實的踩坑經驗和解法、底層邏輯的拆解（為什麼這樣設計而不是那樣）、讀者看完能打開新角度的洞察。

**沒有含金量：** 設定步驟、比較表、「OctoDock 比較好」、各家官方文件已經寫得很清楚的內容。

---

## 三種文章類型

### 類型 A：工具教學文（解決問題導向）

出發點是用戶碰到的具體問題。開頭用「情境」而不是「工具介紹」：

- ✘ 「MCP 是一個開源協議...」
- ✔ 「你啟動了一個長時間任務，離開電腦，然後從手機發一則 Telegram 訊息——Claude 收到後自動處理」

讀者要在 3 秒內知道「這跟我有什麼關係」。

**結構：**

```
# {任務描述}
{結果數字：我每天 15 分鐘處理 60 封信}

## 這篇你會學到
{3-5 個 bullet，每個一句話，讀者掃完就知道值不值得看}

## 你需要什麼
{先決條件：帳號、版本、工具}

## 完整流程
{可複製的 prompt（用 blockquote）}
{每一步加上「做對了的樣子」和「常見錯誤」（用 callout 區分）}

## 幾個你可能不知道的技巧
{3-5 個具體技巧，每個解釋「為什麼」}

## 我踩過的坑
{1-3 個真實問題和解法，用 ⚠️ callout}

## 做不到什麼
{說清楚限制，用 ❌ callout}

## 工具設定
{具體步驟 + 連結，不要一句帶過}
```

Category：依資料庫實際有的選項填寫（先查 schema 確認）

### 類型 B：思考輸出文（擴展認知）

含金量判斷：讀者看完能不能擴展新知、能不能打開他的想法。從個人經驗出發，不是從定義出發：

- ✘ 「MCP 是一個開源標準...」
- ✔ 「我每天用多個 MCP server，這給了我一些對未來的判斷...」

讀者要感受到「這個人真的在用」。

**結構：**

```
# {洞察或判斷}
{一句話說清楚核心觀點}

## 這篇的核心觀點
{3-5 個 bullet，讀者掃完就知道這篇在講什麼}

## 故事 / 發現過程
{怎麼碰到這個問題、怎麼思考的}
{包含「我錯了」的誠實——讀者對這種內容最有共鳴}

## 拆解
{用簡單語言解釋複雜概念}
{用比喻和具體例子，不用術語}
{有判斷、有立場，不是中立列優缺點}

## 延伸思考
{這個洞察能推到哪些其他場景？}

## 結論
{一句話收尾}
```

**素材來源：** 與 Claude 的深度對話中產生的思考和分析、技術架構決策的背後邏輯、產業趨勢觀察和判斷、開發過程中的教訓。

Category：依資料庫實際有的選項填寫（先查 schema 確認）

### 類型 C：產品體驗文（實際使用場景）

從實際使用場景出發，展示 OctoDock 能做什麼、怎麼用：

- ✘ 「OctoDock 支援 Notion、Gmail、GitHub...」
- ✔ 「我今天花了一整天用 OctoDock 掃描整個 codebase，找到 6 個 Dashboard 缺失」

讀者要看到「實際用起來是什麼樣子」。

**結構：**

```
# {具體任務描述}
{一句話說結果}

## 這篇你會看到
{3-5 個 bullet，讓讀者知道接下來的內容}

## 為什麼要做這件事
{任務背景，用故事帶入}

## 實際操作過程
{貼實際的 prompt 和回傳結果}
{讀者要能看到「我就是這樣打一行指令，它就幫我做完了」}
{步驟之間留白，不要黏成一團}

## 碰到的問題
{用 ⚠️ callout，說問題反而更可信}

## 學到的事
{用 💡 callout，操作過程中的發現和技巧}

## 想試試嗎？
{OctoDock 註冊連結 + 需要連哪些 App}
```

**素材來源：** AI 日常使用 OctoDock 的真實體驗、具體透過 OctoDock 執行的任務紀錄、使用中發現的問題和優化過程。

Category：依資料庫實際有的選項填寫（先查 schema 確認）

## Claude 專欄

聖堯是 Claude 重度使用者（claude.ai、Claude Code、Cowork、Skills）。Blog 應包含 Claude 相關內容。

### 寫作規範

**必須先爬官網再寫，不能用過時的記憶：**

```
web_search("{主題} site:docs.anthropic.com OR site:code.claude.com OR site:claude.com")
web_fetch("https://docs.anthropic.com/...")  // API 文件
web_fetch("https://code.claude.com/...")     // Claude Code 文件
web_fetch("https://claude.com/...")          // 產品頁、Blog
web_fetch("https://support.claude.com/...")  // 幫助中心
```

以當下查到的官方文件為準。如果記憶裡的資訊跟官方文件不一致，以官方文件為準。

### 可寫的主題方向

不寫死功能名，每次寫 Claude 文章前先查最新功能：

```
web_search("Claude new features site:docs.anthropic.com OR site:code.claude.com")
web_fetch("https://docs.anthropic.com/...")
web_fetch("https://code.claude.com/...")
```

從查到的最新功能中找選題，方向包括：新功能介紹、使用指南、最佳實踐、功能比較、更新速報。

### Claude 文章的類型判斷

- 教怎麼設定某個 Claude 功能 → 類型 A（工具教學）
- 分析某個 Claude 功能的本質和未來影響 → 類型 B（思考輸出）
- 記錄一整天用 Claude + OctoDock 做了什麼 → 類型 C（產品體驗）

## 流程

### -1. 同步 Skill（每輪第一次必做，同一輪只做一次）

這份 skill 檔案可能不是最新的。Notion 上的「Blog Skill」頁面（intent-matrix-blog）才是 single source of truth，聖堯會直接在那裡修改規則。

**每輪寫文章的第一篇之前，先拉 Notion 最新版比對：**

```
octodock_do(app: "notion", action: "get_page", params: {
  page_id: "32ba9617-875f-81a4-a751-cbc0b7668487"
})
```

如果回傳被截斷，用 `get_stored` 取完整內容：
```
octodock_do(app: "system", action: "get_stored", params: { ref: "{ref_id}" })
```

拿到內容後，跟這份 `.claude/skills/blog-writer.md` 比對：
- **有差異** → 以 Notion 為準，自動更新本地 skill 檔案（用 Edit 工具寫入），然後繼續流程
- **無差異** → 跳過，繼續流程

**規則：**
- 同一輪對話內只做一次同步，不用每寫一篇都拉
- 如果 Notion 頁面讀取失敗，用本地版本繼續，不要卡住
- 更新後不需要 commit，下次對話會再同步

### 0. 取矩陣維度

```
octodock_help()  → 當前支援的 App
query_database(328a9617-875f-81f5-8923-c66e691c57d0) → 已有文章
```

### 1. 選主題

先判斷文章類型：
- 用戶想解決一個具體問題 → 類型 A
- 有一個洞察或判斷想分享 → 類型 B
- 有一個真實使用場景值得記錄 → 類型 C

**選題優先級：** 搜尋量（Google autocomplete 粗判）> 能實測出真實技巧的 > 中英各半。

**任務類別（持續擴充）：** 信件管理、會議與行程、專案管理、內容與文件、開發者工作流、AI 工具設定與整合、Claude 使用技巧。

**App 選題矩陣產生方法：**

選題不寫死在 SKILL 裡。每次要產文章時，動態查最新的 App 和 action，再思考選題。

步驟：

1. 查 OctoDock 目前有哪些 App 和 action：
   ```
   octodock_help()  → 看所有已連結的 App
   octodock_help(app: "{app_name}")  → 看該 App 的所有 action
   ```

2. 查已有文章避免重複：
   ```
   query_database(328a9617-875f-81f5-8923-c66e691c57d0) → 已發佈和 draft 的文章
   ```

3. 從三層結構思考選題：
   - **單一 action 場景**：這個 action 單獨能解決什麼日常問題？
   - **同 App 內 action 搭配**：哪幾個 action 串起來能完成一個完整流程？
   - **跨 App 搭配**：哪兩三個 App 的 action 組合起來能做到單一 App 做不到的事？

4. 選題判斷標準：
   - 這個場景是使用者真的會碰到的嗎？
   - 用 OctoDock 做跟手動做差別大嗎？
   - 跨 App 場景優先（這是 OctoDock 的核心差異化）
   - 不要硬湊不自然的組合

5. 選好的題目寫入 Notion Blog 資料庫（Status: draft）

### 2. 查證

**2a. 看別人怎麼寫同類文章（每篇必做）**

寫之前先搜同主題的文章，看排版、角度、結構：

```
web_search("{文章主題關鍵字} tutorial")
web_search("{文章主題關鍵字} how to")
```

挑 2-3 篇讀完，觀察：
- 別人用什麼角度切入？有沒有我沒想到的？
- 排版怎麼做的？段落長度、用了什麼視覺元素？
- 讀者留言或回饋裡反映了什麼需求？
- 哪些寫法讀起來舒服？哪些讀起來像 AI 寫的？

不是抄，是知道「市場上已經有什麼」再寫出不一樣的。

**2b. 查 AI 工具內建 connector（最容易漏）**

搜 `{AI工具} {App} connector built-in` 或直接查 connector 頁面。有內建的話文章裡要提到。

**2c. 查工作流可行性**

用 OctoDock 實測完整流程。文章裡的 prompt 要真的跑過。踩坑經驗來自實測或社群（Reddit、Discord），不編造。

每個踩坑經驗必須標註來源：
- 🔬 **實測**：自己跑過、確認會發生的
- 🔍 **搜尋**：從別人的文章、社群討論看到的（附來源連結）
- 🤔 **推測**：根據 API 文件或經驗推斷的，還沒實測

文章裡只放「實測」和「搜尋」的踩坑。「推測」的不放進文章，但記錄到 Blog Skill 回饋頁面，之後找機會驗證。

**2d. 查 OctoDock 步驟**

`web_fetch("https://octo-dock.com/docs")` 確認操作步驟是最新的。

**2e. Claude 相關文章額外查證**

如果文章涉及 Claude 功能，必須先爬官網確認最新資訊（見 Claude 專欄寫作規範）。

### 3. 寫文章

根據判斷的類型，使用對應的結構（見上方三種類型）。

**寫入 Notion：**

先查資料庫 schema 確認欄位名稱和選項：

```
octodock_do(app: "notion", action: "query_database", params: {
  database: "328a9617-875f-81f5-8923-c66e691c57d0",
  page_size: 1
})
```

再根據查到的欄位寫入：

```
octodock_do(app: "notion", action: "create_database_item", params: {
  database: "328a9617-875f-81f5-8923-c66e691c57d0",
  properties: { 根據查到的 schema 填寫，Status 一律 draft },
  content: "Markdown"
})
```

單頁上限 100 blocks，太長要精簡。

### 4. 用戶視角驗收（每篇必檢）

**類型 A 驗收清單：**
- [ ] 用戶碰到什麼問題？→ 寫在開頭
- [ ] 他可能搜什麼？→ 決定 title 和 SEO keyword
- [ ] 他想知道什麼？→ 決定內容結構
- [ ] 實際上能不能解決？→ 說清楚能力和限制

**類型 B 驗收清單：**
- [ ] 讀者看完能不能學到一個之前不知道的概念？
- [ ] 能不能打開他對某個問題的新角度？
- [ ] 有沒有具體的「啊哈！」時刻？

**類型 C 驗收清單：**
- [ ] 讀者看完知不知道「實際用起來是什麼樣子」？
- [ ] 有沒有真實的呼叫範例和回傳結果？
- [ ] 有沒有說問題（不只說好話）？

### 5. 自我審查 + 優化

每篇寫完後用三個標準評分：

1. **有沒有讀者想不到的東西？** 如果整篇都是「搜一下就知道」的內容 → 砍掉重寫或補技巧。
2. **會不會想看完？** 如果開頭沒有畫面感、中間沒有「原來如此」的時刻 → 改開頭場景、加順序邏輯解釋。
3. **工具推薦自然嗎？** 如果讀者會覺得「這是一篇賣東西的文」→ 減少推薦段落、讓工具只在設定段出現。

每篇給 1-5 星。3 星以下不發，回去補含金量。

**審查後修改文章內容（replace_content），不是只在回報裡寫「要改」。**

### 6. 回饋優化 skill（寫到 Notion）

每批文章產完後，把回饋 append 到 Notion「Blog Skill 回饋」頁面：

```
octodock_do(app: "notion", action: "append_content", params: {
  page_id: "32ba9617-875f-81e4-84e4-c36be36a4d1a",
  content: "## {日期} 第 N 批回饋\n\n### 新發現的技巧\n{實測中發現的新技巧、新場景}\n\n### 寫法調整\n{哪些寫法讀者可能不買單、該怎麼改}\n\n### 查證流程補充\n{哪些查證步驟漏了}\n\n### 新選題靈感\n{過程中想到的新選題}"
})
```

**注意：不要只在對話裡講回饋，一定要寫到 Notion。** 聖堯會定期根據這裡的內容更新 SKILL.md。

### 7. 回報（在對話裡給聖堯看）

```
## 本批結果

### 文章 + 自審
| 標題 | 類型 | 含金量來源 | 評分 | 優化了什麼 |
（每篇一行）

### 實測發現
（新技巧、新坑、新任務場景）

### 進度
本次 / 總篇數 / 建議下一批
```

回饋細節已寫到 Notion（步驟 6），這裡只放摘要讓聖堯快速掌握。

## 排版指引

### 呼吸感
- 每 2-3 段之間要有空行或視覺分隔（分隔線或小標題），讀者的眼睛需要休息
- 一段不超過 4-5 行，太長就拆
- 重要段落前後留白，不要黏在一起

### 粗體要節制
- 一個段落最多一處粗體
- 粗體用在「讀者掃讀時最該注意的那句話」，不是用在每個段落標題

### Prompt 和程式碼
- 可複製的 prompt 用 blockquote（>）
- 程式碼或技術指令用 code block
- prompt 前加一句「你可以直接複製這段：」，降低使用門檻

### 提示框（Notion callout block）

在 Notion 裡用 callout block 區分不同類型的提示：
- ⚠️ **踩坑警告**：容易出錯的地方
- 💡 **技巧**：讀者想不到的小撇步
- ❌ **做不到的事**：誠實說限制
- ✅ **成功的樣子**：讓讀者知道做對了長什麼樣

不要把這些埋在段落裡，要用 callout 讓它們在視覺上跳出來。

### Key Takeaways

每篇文章開頭（標題和第一段之後）放 Key Takeaways：
- 3-5 個 bullet points
- 每個一句話
- 讀者只看這段就知道這篇值不值得往下讀

### 工具設定段

- 附上 OctoDock 註冊連結
- 說明需要連接哪些 App
- 如果有其他方式（例如 Claude 內建 connector），也提到差異
- 不要硬推銷，說事實就好

### 口語化

- 寫給「下班後想用 AI 提升效率的上班族」看的，不是寫給工程師看的
- 用「你」不用「使用者」
- 用「試試看」不用「執行」
- 用「會出錯」不用「將導致錯誤」
- 技術術語第一次出現時加一句白話解釋

### 已發佈文章的排版修正

如果聖堯要求更新舊文章，對照以上規則逐條檢查，用 replace_content 修正。

## 寫作鐵律

1. 標題用使用者語言，不提工具名
2. 每篇至少一個讀者想不到的東西，沒有就不發
3. 可直接複製的 prompt（用 blockquote）
4. 第一段是結果數字或情境，不是背景介紹
5. 第一段之後馬上放 Key Takeaways（3-5 bullet）
6. 踩坑經驗必須標註來源（🔬實測 / 🔍搜尋），推測的不放進文章
7. 用 callout 區分提示類型（⚠️ 警告 / 💡 技巧 / ❌ 限制 / ✅ 成功）
8. 一段不超過 4-5 行，段落之間留白
9. 粗體一段最多一處，不要每段開頭都粗體
10. 工具設定放最後，但要附連結和具體步驟，不要一句帶過
11. 口語化：用「你」不用「使用者」，用「試試看」不用「執行」
12. 每篇寫完自我審查，3 星以下不發
13. Status 一律 draft
14. 每批產完回饋寫到 Notion
