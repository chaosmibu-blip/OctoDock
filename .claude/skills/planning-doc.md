# 規劃文件要求

## 什麼時候觸發

當一個任務涉及**修改 3 個以上的檔案**時，必須先建立規劃文件再動手。

觸發情境包括但不限於：
- 用戶要求一個新功能，需要動到多個模組
- Bug 修復牽涉到多個檔案的連動
- 架構重構或重命名
- 新增一個完整的子系統（例如新增 App adapter + 路由 + DB schema）
- 任何你判斷「這不是改一兩行就能搞定」的任務

## 規劃文件要包含什麼

建立 `docs/plan-{簡短描述}.md`，內容：

### 1. 目標
一句話說明這次要做什麼、為什麼做。

### 2. 影響範圍
列出所有會被修改的檔案，每個檔案寫一行說明改什麼。

### 3. 執行步驟
按照依賴順序排列，標明哪些可以並行。

### 4. 驗證方式
怎麼確認改完是對的（build 通過、測試 case、手動測試）。

### 5. 風險
有沒有破壞性變更？需不需要 DB migration？會不會影響現有用戶？

## 流程

1. 先寫規劃文件
2. 給用戶看（或在對話中簡述重點）
3. 用戶確認後才開始寫程式碼
4. 完成後把規劃文件中的狀態標記為「已完成」
5. 如果規劃文件已經不再需要（任務完成且內容已反映在程式碼中），刪掉規劃文件避免堆積

## 例外

以下情況不需要規劃文件：
- 單一檔案的 bug fix
- 只改 2 個以下檔案的小修改
- 用戶明確說「直接做」
