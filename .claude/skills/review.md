# 審查

> 審查不是找碴，是用不同的視角重新看一遍同樣的東西。
> 開發時你在「怎麼做」，審查時你在問「做對了嗎」「漏了什麼」「會壞在哪」。

### 職責邊界

- **負責**：確認產出是否正確、完整、合理。從五個維度（程式碼、架構、設計、產品、安全）檢視
- **不負責**：修問題（→ 開發）、分析根因（→ 思考）、長期健康（→ 維護）
- **產出**：問題清單，分三級 — 必修（阻塞上線）、建議（改了更好）、觀察（值得記錄）
- **從哪來**：開發完成的程式碼、定期稽核排程
- **往哪去**：必修項回到開發修、觀察項進維護追蹤、系統性問題交給思考

---

## 什麼時候審查

| 時機 | 審查哪些維度 |
|------|------------|
| 開發完成、提交前 | 程式碼 + 安全 |
| 新功能上線前 | 全部五個維度 |
| 改了核心架構 | 架構 + 程式碼 + 安全 |
| 改了 UI/UX | 設計 + 產品 |
| 定期稽核 | 全部五個維度 |
| 用戶回報問題後修復 | 程式碼 + 產品 |

## 一、程式碼審查

### 正確性

- 邏輯是否正確？邊界條件有沒有處理？
- 錯誤路徑有沒有被覆蓋？每個 catch 都做了正確的事？
- 資料流完整嗎？每個欄位從輸入到寫入到讀取，路徑上有沒有斷裂？

### 完整性

- 改了一處，所有相關的地方都改了嗎？（grep 驗證）
- 前後端同步了嗎？（APP_KEYS、i18n、oauth-env）
- DB migration 有沒有處理既有資料？
- 所有失敗路徑都有 logOperation 嗎？

### 品質

- **命名即邏輯**：精準的命名讓程式碼自解釋，錯誤的命名會建立錯誤的心智模型
- **精準描述**：精準的描述可以避免未來的誤解。註解補充命名無法表達的「為什麼」
- 有沒有不必要的複雜度？（過早抽象、多餘的配置、沒人用的參數）
- 重複程式碼出現 2 次以上了嗎？該抽共用嗎？
- 單一元件/函式超過 300 行了嗎？

### 註解完整性

- **每個函式**都有註解說明用途、參數意義、回傳值嗎？
- **複雜邏輯**有說明「為什麼這樣做」嗎？（不是重述程式碼，是解釋決策）
- **分支條件**有說明觸發情境嗎？（什麼情況會走到這個 if）
- **註解跟實作一致嗎？** 改了程式碼沒改註解 = 比沒有註解更糟
- 發現漏註解或描述不清的 → 補上，不是觀察項，是必修項

### 可維護性

- 下一個人讀這段程式碼，能在 30 秒內理解意圖嗎？
- 錯誤訊息說人話嗎？（「Token 無效，請從 @BotFather 重新複製」不是「Error 401」）

---

## 二、架構審查

### 層次正確性

- 這個邏輯放對層了嗎？（各層職責對照表見 `thinking.md` 架構決策章節）
- 通用邏輯有沒有寫在個別 adapter 裡？
- App 專屬邏輯有沒有寫在通用層？

### 擴展性

- 這個設計在第二個 App 出現同樣需求時，能不能直接用？
- 同一個問題在第二個 App 出現 → 該提升到架構層了（決策流程見 `thinking.md`）
- 有沒有硬編碼只適用於特定 App 的邏輯在通用層？

### 一致性

- 新程式碼跟現有架構的慣例一致嗎？
- TypeScript interface 的 required/optional 設對了嗎？
- 新增的 App 有沒有通過 isAppAdapter type guard？

---

## 三、設計審查

### UI/UX

- 零學習成本：打開就知道怎麼用嗎？
- 最少步驟：能 2 步完成的事有沒有設計成 5 步？
- 五種狀態都處理了嗎？（正常/空/載入中/錯誤/邊界值）
- 響應式：手機、平板、桌面都正常？觸控目標 ≥ 44px？
- 錯誤回饋即時且有用？不靜默？

### 資訊架構

- 用戶最需要的資訊在最顯眼的位置嗎？
- 漸進式揭露：預設只顯示最重要的，細節按需展開？
- 導航邏輯清楚嗎？用戶知道自己在哪、能去哪？

---

## 四、產品審查

### 用戶價值

- 這個功能解決了用戶的什麼問題？
- 用戶會怎麼發現和使用這個功能？
- **技術正確 ≠ 有用**：統計上正確但實際有害的功能比不做更糟

### 邊界情境

- 新用戶（沒有任何資料）看到什麼？
- 重度用戶（大量資料）效能還行嗎？
- 用戶做錯了怎麼辦？能撤銷嗎？錯誤提示能引導修正嗎？

### 引導流程

- 用戶的瓶頸在哪一步？引導有沒有解決那一步？
- 做完一個動作後，下一步建議合理嗎？（不建議破壞性操作）

---

## 五、安全審查

### 認證與授權

- Token 有沒有明文出現在日誌、回應、錯誤訊息中？
- OAuth refresh token 有沒有正確存儲和使用？
- 用戶只能操作自己的資料嗎？

### 輸入驗證

- 所有外部輸入（用戶參數、API 回傳）都有驗證嗎？
- 有沒有 SQL injection、XSS、command injection 風險？
- 檔案上傳有限制大小和類型嗎？

### 資料保護

- 敏感資料有加密嗎？
- 資料保留政策有遵守嗎？（operations 90 天、stored_results 24 小時）
- 第三方 API 的回傳有沒有包含不該暴露的資訊？

---

## 審查輸出

審查完畢後，產出的不是「通過/不通過」，而是：

1. **必修** — 不改就不能上線的問題（安全漏洞、資料流斷裂、邏輯錯誤）
2. **建議** — 改了會更好但不阻塞上線（命名優化、程式碼簡化）
3. **觀察** — 不需要現在改但值得記錄（潛在的擴展性問題、技術債）

### 審查後的行為

- **有必修項 → 報告完直接進入修復**，不等用戶推。審查和開發是同一人時，交接應無感
- 建議項和觀察項只報告，由用戶決定是否處理
