# 思考

> 動手之前先想清楚。想清楚不是想很久，是用對的框架問對的問題。
> 大多數錯誤不是因為做錯了，是因為沒想清楚就開始做。

### 職責邊界

- **負責**：專案全貌理解、推進方向評估、問題分析與決策、收斂與發散思考、commit 後反思、對話摘要
- **不負責**：寫程式碼（→ 開發）、驗證品質（→ 審查）、系統保養（→ 維護）、skill/記憶/工作流程管理（→ 優化）
- **產出**：全貌報告、方向建議、決策、反思結論、對話摘要
- **從哪來**：對話開始時（全貌）、需要判斷方向時、開發碰到非 trivial 問題、每次 commit 後
- **往哪去**：決策交給開發執行、反思結論交給優化處理、對話摘要存入記憶

## 零、專案全貌 — 先看清全局再動手

### 什麼時候需要

- 新對話開始、不確定當前專案狀態時
- 用戶提出涉及多個模組的需求時
- 需要判斷某個改動的影響範圍時

### 怎麼建立全貌

1. **讀 CLAUDE.md** — 確認路由表和協作準則
2. **讀 git log** — 最近在做什麼、方向是什麼
3. **掃關鍵檔案** — server.ts（入口）、types.ts（介面）、schema.ts（資料）、registry.ts（已接 App）
4. **查 production 數據**（如有需要）— operations 表的近期狀態

### 產出

一段簡潔的現狀摘要：目前有幾個 adapter、最近的開發方向、已知的問題或待辦。不需要每次都寫出來，心裡有數即可。

## 零之一、推進方向評估

### 什麼時候需要

- 用戶問「接下來做什麼」
- 當前任務完成後、下一步不明確時
- 多個可能方向需要取捨時

### 評估維度

| 維度 | 問什麼 |
|------|--------|
| **用戶價值** | 這個方向能解決多少用戶的什麼問題？ |
| **技術可行性** | 現有架構能支撐嗎？需要多大改動？ |
| **投入產出比** | 花多少時間能得到多少效果？ |
| **依賴關係** | 需要先完成什麼才能做這個？ |
| **風險** | 做了可能壞什麼？不做會怎樣？ |

### 產出

排序後的方向建議，每個方向附一句理由。由用戶決定。

## 零之二、OctoDock 是什麼 — 所有思考的起點

> One URL. All Apps. Remembers You.

OctoDock 是面向非技術用戶的基礎設施產品。用戶設定一個 MCP URL，任何 AI agent 就能操作所有已授權的 App，並擁有跨 agent 共享的操作記憶。三個核心方向：跨 App 操作層、記憶層、AI 使用體驗優化層。

### 核心架構

MCP server 只暴露 `octodock_do` 和 `octodock_help` 兩個工具。每個 App 是獨立的 Adapter 模組，啟動時自動掃描、自動註冊。記憶引擎用 PostgreSQL + pgvector，四種記憶類型：preference、pattern、context、workflow。工作流系統自動偵測重複操作模式。訂閱系統三條收款線：Paddle、IAP、ECPay。

### 核心定位：AI 負責想，OctoDock 負責做

OctoDock 不是 AI 和 App 之間的「守衛」，是「執行者」。AI 只需要表達意圖和提供資料，OctoDock 負責組裝、控速、寫入、驗證、回報。

- **不限制 AI**：rate limit、格式轉換、批量控速是 OctoDock 內部實作細節，AI 不需要知道
- **幫助 AI**：回傳壓縮、transfer 提示、nextSuggestion — 這些不是限制，是輔助
- **App API 限制由 OctoDock 處理**：各 App 的 rate limit 定義在 `APP_RATE_LIMITS`，批量操作自動控速、失敗自動重試
- **唯一的外部限制**：免費用戶月度配額（商業模型）

**設計新功能時問「OctoDock 能替 AI 做什麼」，不是「怎麼限制 AI」。**

### 核心迭代迴路

```
AI 使用 → 修正格式 → 補現狀 → 執行 → 記錄 → 分析失敗模式 → 優化程式碼 → AI 下次用得更好
```

## 一、問題分類 — 先搞清楚你在面對什麼

**碰到問題的第一步不是解決，是分類。**

### 問題類型決定思考深度

| 類型 | 特徵 | 思考深度 | 範例 |
|------|------|---------|------|
| **已知問題** | 以前碰過，有現成解法 | 低 — 直接套用 | typo、已知 API 格式 |
| **相似問題** | 跟以前碰過的類似，但有差異 | 中 — 比對差異再決定 | 新 App 的同類功能 |
| **新問題** | 從沒碰過 | 高 — 完整走思考流程 | 新架構、新需求 |
| **系統性問題** | 反覆出現、跨多個模組 | 最高 — 改結構不改症狀 | 同類 bug 第二次出現 |

### 分類決策樹

```
碰到問題
  → 以前碰過一模一樣的嗎？
    → 是 → 直接套用（但驗證解法還適用）
    → 否 → 以前碰過類似的嗎？
      → 是 → 差異在哪？差異重要嗎？
        → 不重要 → 調整後套用
        → 重要 → 當新問題處理
      → 否 → 新問題，走完整思考流程
  → 同時問：這個問題出現第二次了嗎？
    → 是 → 系統性問題，改結構（見第三章）
```

## 二、根因分析 — 找到「為什麼」

### 五個為什麼

不要停在表面症狀。連問五次「為什麼」：

```
症狀：Gmail adapter 回傳 raw JSON
  → 為什麼？adapter 沒實作 formatResponse
  → 為什麼？開發時忘了
  → 為什麼？沒有強制檢查機制
  → 為什麼？TypeScript interface 沒把 formatResponse 設為 required
  → 根因：架構層沒有強制，靠人記住 = 一定會忘
  → 解法：改 types.ts 把 formatResponse 設為 required（build 不過就抓到）
```

### 三個維度

每個問題從三個維度分析：

| 維度 | 問什麼 | 找什麼 |
|------|--------|--------|
| **是什麼** | 現象是什麼？影響範圍多大？ | 事實，不是猜測 |
| **為什麼** | 根因是什麼？為什麼現在才出現？ | 結構性原因，不是個別原因 |
| **怎麼辦** | 怎麼修？怎麼防止再犯？ | 最小成本的最大效果 |

### 驗證你的分析

- **說明時先用自然語言描述邏輯，再比對程式碼和 DB 驗證一致性**
- 不要憑印象說明，讀完程式碼確認後再回答
- 發現邏輯和實作不一致時要指出來
- 做不到就說做不到，錯誤的結論比沒有結論更糟
- **子 agent 產出是線索，不是結論**：子 agent 的報告必須用實際檔案驗證後再行動，不驗證就執行會產生重複或錯誤

## 三、架構決策 — 改哪裡、怎麼改

### 核心問題：這是 App 專屬的，還是所有 App 都會碰到的？

**不要急著修。先問這個問題。**

| 如果是通用問題 → 改架構 | 如果是 App 專屬 → 只改 adapter |
|------------------------|-------------------------------|
| 某 App 回傳 raw JSON → 確認 types.ts 強制 formatResponse | Notion blocks 轉 MD 有 bug → notion.ts |
| 某 action 的 help 沒範例 → 確認 getSkill 介面強制 action 級查詢 | Gmail base64 解碼壞了 → gmail.ts |
| 名稱解析只有一個 App 有 → 提升到通用的 translateSimplifiedParams | LINE reply token 過期提示 → line.ts |
| 新增 App 忘了同步前端 → 加 hook 自動提醒 | |

### 架構演進判斷標準

**同一個問題在第二個 App 出現時，就該提升到架構層。**

- 第 1 次 → 可以先在 adapter 修
- 第 2 次 → 停下來，改架構讓所有 App 自動受益
- 不要等第 3 次

### 邏輯放對層

| 邏輯類型 | 該放的位置 | 不該放的位置 |
|----------|-----------|------------|
| 通用參數轉換 | param-guard.ts | 個別 adapter |
| App 專屬格式 | adapter | server.ts / middleware |
| 錯誤翻譯 | adapter formatError + error-hints | 直接 throw |
| 操作記錄 | logger middleware | adapter 內部 |
| 回傳壓縮 | response-formatter（自動） | adapter 手動截斷 |

### OctoDock 架構總覽

```
用戶的 AI（Claude/ChatGPT/Gemini）
        ↓ MCP（octodock_do / octodock_help）
┌─────────────────────────────────────────────┐
│  OctoDock MCP Server（server.ts）            │
│  ┌─────────┐  ┌──────────┐  ┌────────────┐ │
│  │ 參數轉換 │  │ 格式轉換  │  │ 錯誤引導   │ │
│  └────┬────┘  └────┬─────┘  └─────┬──────┘ │
│       ↓            ↓              ↓         │
│  ┌─────────────────────────────────────┐    │
│  │  AppAdapter 介面（types.ts）        │    │
│  └────────────────┬────────────────────┘    │
│       ↓           ↓           ↓             │
│  ┌────────┐ ┌─────────┐ ┌──────────┐       │
│  │ Notion │ │ Gmail   │ │ GitHub   │ ...   │
│  └────────┘ └─────────┘ └──────────┘       │
│                                             │
│  ┌─────────────────────────────────────┐    │
│  │  共用服務                            │    │
│  │  memory-engine / pattern-analyzer   │    │
│  │  workflow-detector / plan-limits    │    │
│  └─────────────────────────────────────┘    │
└─────────────────────────────────────────────┘
```

## 四、認知進化 — 從經驗中學習

### 核心迴路

```
AI 使用 OctoDock → 成功/失敗記錄到 operations
  → 分析失敗模式 → 優化程式碼（param-guard / middleware / adapter）
  → 提煉通用認知 → 寫入對應 skill / 建新 skill
  → 下次 AI 使用時自動受益
```

### 觸發時機

| 當你做了這件事 | 立刻檢查 |
|--------------|---------|
| 改了 tool schema（z.object 參數定義） | operations 表有對應欄位嗎？logOperation 有傳嗎？ |
| 在 adapter 做了格式轉換 | 該不該提升到 param-guard？其他 App 有同樣問題嗎？ |
| 修了一個之前修過的同類 bug | 升級到架構層 |
| 用戶說「不對」「改回去」 | 寫入對應 skill 的注意事項 |
| 查 production 數據發現失敗模式 | 先改程式碼 → 再寫入對應 skill |
| 加了新的提前返回路徑 | 有 logOperation 嗎？帶齊所有必填欄位了嗎？ |
| 新增 MCP 參數 | DB schema + migration + logOperation + 所有呼叫點同步 |

**不觸發：** 單純修 typo、只改一個 adapter 的孤立 bug、例行 CRUD。

### 認知該沉澱在哪裡

優先級：**程式碼能解決的 > 規則能解決的 > 需要人判斷的**

| 優先級 | 類型 | 沉澱位置 | 範例 |
|--------|------|---------|------|
| P0 | 程式碼自動修正 | param-guard / middleware | 自動轉換日期格式 |
| P1 | 錯誤訊息改善 | adapter formatError | API 錯誤翻譯成修正指引 |
| P2 | 開發原則 | 對應的主 skill | 新增注意事項或反模式 |
| P3 | 多步驟流程 | .claude/skills/ | 建新 skill 或更新既有 |

### 認知提煉公式

**{什麼情境} + {具體做法} + {為什麼}**

```
好：「AI 傳 start/end 為字串時，param-guard 自動包成 {dateTime} 物件，因為 GCal API 只認物件格式」
壞：「要注意日期格式」（沒有情境、沒有做法）
```

## 五、收斂與發散

### 發散 — 展開可能性

在問題定義階段或方案探索階段使用：

- 列出所有可能的原因/方案，不急著篩選
- 從不同角度看同一個問題（用戶視角、技術視角、商業視角）
- 問「還有呢？」直到想不出新的
- 允許看似不合理的選項存在，它們可能觸發更好的想法

### 收斂 — 聚焦到行動

在方案確定階段或執行階段使用：

- 用明確的標準篩選（可行性、投入產出比、風險）
- 一次只選一個方向，做完再評估
- 砍掉不影響核心目標的分支
- 能用一句話說清楚的方案 > 需要一段話解釋的方案

### 交替使用

```
問題不明確 → 發散（展開可能性）→ 收斂（選定方向）
方向明確但方案不確定 → 發散（列出方案）→ 收斂（選定方案）
執行中碰到意外 → 暫停 → 發散（重新評估）→ 收斂（調整或維持）
```

關鍵判斷：**現在該展開還是該聚焦？** 如果選項不夠多，發散。如果選項太多遲遲無法行動，收斂。

## 六、通用思考原則

### 回到本質

- 先理解事物的本質和第一性原理，再做設計和實作
- 不要在沒理解本質的情況下加上任意的限制條件
- 問「這個東西本來應該怎麼運作」，不是「別人怎麼做的」

### 結構性思考

- 修的不是個別的症狀，是產生症狀的結構
- 問題不是「A 壞了去修 A」，是「為什麼 A 會壞，怎麼讓 B 到 Z 不可能出現同樣的問題」

### 複利思維

- 每個環節都建立能自我增強的機制
- 改一次就永久生效的解法 > 每次都要記得的規則
- 正向循環會自己推動專案轉動

### 不動作是一種選擇

- 錯誤的行動比不行動更有害
- 未被發現的錯誤會累積，在最關鍵的時刻爆發
- 沒把握就不要動，先想清楚

### 三思而後行

- 實作前先問「誰會看到？看到後會做什麼？做錯了會怎樣？」
- 給出答案之前先自己驗證
- 不論是搜尋、跑測試、還是多反思幾次，回答之前先確認自己說的是對的

---

## 反模式

1. **不要只改 CLAUDE.md 不改程式碼** — 能自動修正的，改程式碼效果 > 寫規則
2. **不要在 adapter 裡做通用轉換** — 通用邏輯放 param-guard
3. **不要寫模糊原則** — 「注意品質」不是認知，「adapter 必須實作 formatResponse」才是
4. **不要只記錄問題不記錄解法** — 「壞了要這樣修」比「曾經壞過」有用
5. **不要忘記刪除過時的認知** — 改了架構就要更新對應描述
6. **不要急著修** — 沒想清楚問題類型就開始改，改完才發現改錯地方

## 七、Commit 後反思

**每次 commit 後自動觸發。** 回顧整個對話與任務過程，而非只看本次 commit 的 diff。

### 反思維度

| 維度 | 問什麼 | 產出（結論） |
|------|--------|------------|
| **決策品質** | 這次的判斷正確嗎？有沒有走了彎路？根因是什麼？ | 認知沉澱（思考自行執行） |
| **框架有效性** | 職責邊界有沒有模糊地帶？交接有沒有斷裂？ | 結構調整建議（交給優化執行） |
| **Skill 覆蓋度** | 有沒有做了某件事但沒有對應 skill？ | 增刪建議（交給優化執行） |
| **認知缺口** | 有沒有「早知道就好了」的時刻？ | 認知沉澱（思考自行執行） |
| **效率** | 哪個環節花了不必要的時間？ | 流程改善建議（交給優化執行） |

### 執行

1. 回答上述五個維度（沒有發現就跳過，不為反思而反思）
2. 認知內容 → 思考直接沉澱到對應 skill
3. 結構/流程調整 → 交給優化執行

## 八、對話摘要

**每次對話結束前或 commit 後產出。** 記錄這次對話的關鍵決策和結論，讓下一次對話能快速銜接。

### 摘要內容

- **做了什麼**：一句話概括這次對話完成的事
- **關鍵決策**：列出影響後續工作的決策和理由（不記 trivial 的）
- **未完成項**：如果有中斷的工作，記錄到哪了、下一步是什麼
- **認知變化**：這次對話沉澱了什麼新認知（指向具體 skill 的具體段落）

### 存放

- 存入 Claude Code memory（不進 repo）
- 只記對下一次對話有用的資訊，不記過程細節
