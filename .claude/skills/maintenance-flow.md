# 維護

> 維護不是等東西壞了才修，是在壞之前就發現。
> 定期維護的目的：讓系統保持在「隨時可以安心改動」的狀態。

### 職責邊界

- **負責**：讓現有系統保持健康。定期檢查、清理、更新，防止退化
- **不負責**：開發新功能（→ 開發）、驗證新程式碼的品質（→ 審查）、分析問題根因（→ 思考）
- **產出**：健康度報告、退化項清單、已完成的清理和更新
- **從哪來**：排程觸發、審查產出的觀察項（技術債追蹤）
- **往哪去**：發現的問題交給開發修、反覆出現的模式交給思考找根因

## 一、依賴更新

### 檢查項

- 有沒有安全性更新？（`npm audit`）
- 有沒有破壞性變更的大版本升級？
- 鎖定版本（package-lock.json）跟 package.json 一致嗎？

### 執行

1. `npm audit` 檢查已知漏洞
2. `npm outdated` 列出可更新的套件
3. 安全性更新優先處理，功能更新評估影響後再升
4. 更新後 `npm run build` + 跑一遍核心流程驗證

### 注意事項

- 不要一次升所有東西，分批升級方便定位問題
- 破壞性變更要讀 changelog，確認影響範圍

## 二、技術債清理

### 檢查項

- 累積的 TODO 和 FIXME（`grep -r "TODO\|FIXME" src/`）
- 已知但未修的問題
- 已廢棄但沒刪的程式碼（dead code）
- 過時的註解（描述跟程式碼不一致）

### 執行

1. 掃描所有 TODO/FIXME，評估哪些該修、哪些該刪
2. 已完成功能的暫時寫法（workaround）→ 改成正式寫法
3. 沒有任何地方引用的 export → 刪除
4. 過時的型別定義或 interface → 更新或刪除

### 注意事項

- 不是所有技術債都要還。評估：修的成本 vs 不修的風險
- 清理時順手跑審查（`review.md`），不要只刪不查

## 三、資料衛生與完整性稽核

### 資料保留

| 資料 | 保留政策 | 檢查方式 |
|------|---------|---------|
| stored_results | 24 小時過期 | 查有沒有過期未清的 |
| operations | 90 天保留 | 查有沒有超齡資料 |
| 記憶（memories） | 長期保留 | 查有沒有重複或過時的 |
| workflow | 長期保留 | 查有沒有不再使用的流程 |

### 資料完整性稽核（重大改動後或定期稽核）

1. 欄位填充率：有沒有應該有值但全空的欄位？
2. 資料流完整性：schema 定義的欄位，程式碼有沒有實際寫入？
3. 跨表一致性：關聯的資料有沒有孤兒記錄？
4. 發現問題 → 追溯到程式碼的哪條路徑漏了 → 修復 → 更新規則

### 執行

1. 確認自動清理排程正常運作
2. 抽查資料完整性：operations 的必填欄位有沒有 null？
3. 檢查 DB 表大小，有沒有異常膨脹的表
4. 記憶品質：抽樣檢查記憶內容是否準確、有沒有互相矛盾的

## 四、效能監控

### 檢查項

- 有沒有變慢的查詢？（慢查詢日誌）
- DB 連線池使用率正常嗎？
- API 回應時間有沒有明顯退化？
- 記憶體使用量趨勢正常嗎？

### 執行

1. 檢查 operations 表的 duration 欄位，找出 p95 以上的慢操作
2. 確認 DB index 還有效（沒有因為資料量成長而退化）
3. 檢查有沒有 N+1 查詢或不必要的全表掃描
4. 確認 circuit breaker 狀態，有沒有 App 長期在半開狀態

---

## 五、安全稽核

### 檢查項

- 環境變數有沒有明文出現在程式碼或日誌中？
- OAuth token 的 refresh 機制正常運作嗎？
- 有沒有過期或不再使用的 API key？
- 第三方套件有沒有已知漏洞？

### 執行

1. `grep -r "token\|secret\|password\|key" src/` — 排除 false positive 後確認沒有洩漏
2. 檢查 .env.example 跟實際使用的環境變數一致
3. 確認所有 OAuth App 的 scope 是最小必要權限
4. 檢查 CORS、CSP 等安全 header 設定

---

## 六、知識庫維護

### 檢查項

- CLAUDE.md 的描述跟現在的程式碼一致嗎？
- Skills 有沒有過時的、從來沒觸發的？
- i18n 翻譯有沒有缺漏？
- 規劃文件（docs/plan-*.md）有沒有已完成但忘記刪的？

### 執行

1. 讀 CLAUDE.md，比對實際程式碼，更新不一致的描述
2. 掃描 skills，確認觸發條件還有效
3. 檢查 i18n 所有 key 在中英文都有翻譯
4. 刪除已完成的規劃文件

### 注意事項

- 知識庫是給未來的自己和其他開發者看的，過時的比沒有的更有害
- 改了程式碼 → 同步改文件，這是同一個 commit 的事

## 維護排程建議

| 頻率 | 項目 |
|------|------|
| 每次開發後 | 技術債（順手清理）、知識庫（同步更新） |
| 每週 | 依賴更新（安全性）、效能監控 |
| 每月 | 資料衛生、安全稽核、知識庫全面檢查 |
| 每季 | 技術債全面盤點、架構健康度評估 |
