# 開發

> 開發不只是寫程式碼。三個面向：
> **功能開發** — 把需求變成可運作的程式碼（規劃→執行→回顧）
> **API 研究** — 探索好用/可用的第三方 API，擴展 OctoDock 能操作的範圍
> **數據驅動改善** — 用 operations/memory 數據找出 AI 的使用瓶頸，讓 AI 更好地透過 OctoDock 使用各 App

### 職責邊界

- **負責**：功能開發全流程、API 可行性研究、基於 production 數據的系統改善
- **不負責**：判斷該不該做（→ 思考）、驗證做得好不好（→ 審查）、系統長期健康（→ 維護）
- **從哪來**：思考產出的決策和方向、審查發現的問題清單、維護發現的退化項、數據分析發現的改善機會
- **往哪去**：完成後交給審查驗證、學到的東西交給思考沉澱

## 一、規劃

### 1. 理解需求

- 用戶還在說明時**不要急著實作**
- 用自己的話複述一次，確認理解正確
- 問清楚：誰會用？用在什麼場景？成功長什麼樣？

### 2. 研究現狀

- 查 API 規格、技術限制
- **讀現有程式碼**，找可以複用的模組
- grep 所有相關程式碼路徑，不要憑印象

### 3. 定義範圍

| 要定義的 | 具體內容 |
|----------|---------|
| 資料流 | 每個欄位的生命週期：誰產生、誰傳遞、誰寫入、誰讀取 |
| API 契約 | 前後端介面：端點、request/response 格式 |
| UI | 頁面、元件、五種狀態（正常/空/載入中/錯誤/邊界值） |
| 影響範圍 | 新增或修改的欄位，哪些地方會讀、哪些會寫 |
| DB migration | 新增/修改哪些表和欄位，既有資料怎麼處理 |

### 4. 安全性檢查

- Token 有沒有明文暴露？
- 有沒有注入風險（SQL、XSS、command injection）？
- 權限邊界正確嗎？

### 5. 規劃文件

- **3 個以上檔案改動** → 先寫 `docs/plan-{描述}.md`（詳見 `planning-doc.md`）
- 跟用戶確認設計再動手

### 注意事項

- **先理解再動手**：做錯的要回頭刪乾淨，不是往前跑留技術債
- **不擅自省略**：有疑問就問，不要自己決定跳過
- **不動作是一種選擇**：沒把握的設計，不做比做錯好
- **不重複造輪子**：優先看現有的能不能用，改完要刪掉舊的
- **優先使用 OctoDock 自有功能**：操作已連結的 App 用 OctoDock MCP，記憶和工作流用 OctoDock 系統

## 二、執行

### 6. 寫程式碼

- 後端 + 前端 + migration 一起改，不留半成品
- 遵守編碼慣例（詳見 `implementation-rules.md`）
- 改 adapter → 跑品質檢查清單（`adapter-quality-checklist.md`）

### 7. 前後端同步

- 後端改了 → 檢查前端要不要同步（`frontend-sync.md`）
- APP_KEYS、i18n、oauth-env 三個必查

### 8. Build + 驗證

- `npm run build` 通過
- 本地跑一次完整流程：**不是只 build 通過，是實際操作一遍確認行為正確**
- 用實際數據驗證：查 DB 確認欄位填充率符合預期
- **部署提醒**：Replit 不會自動 deploy。build 成功後提醒用戶去 Replit 手動點 Deploy。MCP 呼叫走 production（octo-dock.com），localhost 測試通過不代表 MCP 會生效

### 注意事項

- **改架構時 grep 所有相關程式碼**：改了一處不等於改了全部
- **能自動修正就修正**：不要只回傳 warning 讓 AI 重試
- **不記錄 = 看不見 = 不知道壞了**：每條程式碼路徑都要有觀測手段
- **元件只負責自己知道的**：不知道的交給上層兜底
- **AI 和 API 之間的格式落差是系統性問題**：用架構層統一解決，不是個別 adapter 各做各的

## 三、回顧

### 9. 上線後監控

- 錯誤率、欄位填充率、效能
- 部署到 production 後實際跑一遍

### 10. 完整性檢查

- 改了架構或加了欄位 → grep 所有相關程式碼確認沒有遺漏
- 跑一次審查流程（`review.md`）

### 11. 更新知識

- 更新文件（CLAUDE.md、skills、i18n）
- 學到新東西 → 判斷要不要沉澱（`thinking.md` 的認知進化章節）

### 12. 驗證資料流

- 隔一段時間用 production 數據驗證資料流有沒有斷裂

### 注意事項

- **技術正確 ≠ 有用**：統計上正確但實際有害的功能比不做更糟
- **做不到就說做不到**：錯誤的結果比沒有結果更糟
- **說明時先用自然語言描述邏輯，再比對程式碼和 DB 驗證一致性**

## Bug Fix 捷徑

Bug fix 不需要走完整流程，但核心步驟不能省：

1. **重現**：確認問題存在，釐清觸發條件
2. **定位根因**：不是找到出錯的那一行，是找到**為什麼**會出錯
3. **評估影響範圍**：只影響一個地方，還是同樣問題在其他地方也有？
4. **確認方向**：跟用戶確認修復方向
5. **修復 + 驗證**：確認修復有效，確認沒有引入新問題
6. **回顧**：這個 bug 怎麼產生的？開發流程哪一步漏了？同類 bug 修第二次 → 升級到架構層

---

## Production 事故處理

Production 出問題時，走這個流程而非一般的 Bug Fix 捷徑：

1. **止血**：先讓 production 恢復正常運作，即使是暫時性的 workaround
2. **定位**：找到根因（→ `thinking.md` 根因分析）
3. **修復**：正式修復，移除 workaround
4. **驗證**：確認修復有效
5. **回顧**：為什麼發生？怎麼更早發現？需不需要加監控或警報？

跟 Bug Fix 的差異：事故處理**先止血再找根因**，Bug Fix 先定位根因再修。

## API 研究

為 OctoDock 探索新的第三方 API，擴展 AI 能操作的範圍。

### 什麼時候做

- 用戶要求新增 App
- 發現現有 adapter 的 API 有更好的替代方案
- 探索某個領域（如專案管理、設計工具）有哪些可整合的服務

### 研究流程

1. **搜尋**：用 WebSearch 找 API 規格、定價、rate limit、認證方式
2. **評估可行性**：
   - 有沒有 REST API 或 SDK？
   - 認證方式（OAuth2 / API Key / 其他）是否支援？
   - 免費額度夠不夠日常使用？
   - Rate limit 是否可接受？
3. **評估價值**：
   - 這個 API 能讓 AI 做到什麼之前做不到的事？
   - 目標用戶群有多大？
   - 跟現有 App 有沒有互補性？（如 Google Sheets + Notion = 資料同步場景）
4. **產出**：寫 `docs/{app}-api-reference.md`（詳見 `new-app-research.md`）

### 注意事項

- 不是 API 有就值得接。**問「用戶透過 AI 操作這個 App 的場景是什麼」**，沒有場景就不接
- 優先接用戶已經在用的 App，不接冷門工具

## 數據驅動改善

用 operations 和 memory 資料找出 AI 使用 OctoDock 的瓶頸，針對性改善。

### 數據來源

| 表 | 看什麼 | 能發現什麼 |
|----|--------|-----------|
| operations | 失敗率、失敗原因、duration、action 分佈 | 哪些 action 最常失敗？失敗原因是 AI 參數錯還是 API 限制？ |
| operations | 同一 session 的操作序列 | AI 重試了幾次才成功？哪些 action 的學習曲線最陡？ |
| operations | intent 欄位 | AI 想做什麼 vs 實際做了什麼，落差在哪？ |
| memories | preference / pattern | 用戶的常見操作模式是什麼？有沒有高頻模式可以變成捷徑？ |
| workflows | 多步驟流程 | 哪些流程最常被執行？有沒有可以自動化的？ |

### 改善方向

| 發現 | 對應改善 | 改哪裡 |
|------|---------|--------|
| 某 action 失敗率 > 20% | 分析失敗原因 → 加 param-guard 轉換規則或改善 error hint | param-guard / error-hints |
| AI 對某 action 平均重試 > 2 次 | getSkill 的說明不夠清楚，或參數格式不直覺 | adapter getSkill / param-guard |
| 某兩個 action 總是連續執行 | 考慮合併為 combo 或加強 nextSuggestion | action-chain / combos |
| 某個 App 的 error 類型集中在認證 | OAuth refresh 機制有問題 | adapter refreshToken |
| 用戶反覆手動修正某個偏好 | memory engine 的學習沒生效 | memory-engine / pattern-analyzer |

### 執行流程

1. **查數據**：用 SQL 查 operations 表的失敗模式和使用模式
2. **找瓶頸**：哪個問題影響最多用戶或最高頻？
3. **定方案**：用程式碼解決（P0）> 改善提示（P1）> 加文件（P2）
4. **實作**：走功能開發流程（規劃→執行→回顧）
5. **驗證**：上線後再查同一組數據，確認指標改善
