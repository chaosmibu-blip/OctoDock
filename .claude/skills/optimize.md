# 優化

> 優化不是讓每件事都完美，是找到瓶頸然後消除它。
> 問的不是「怎麼做得更好」，而是「哪裡最痛？痛的根因是什麼？」

### 職責邊界

- **負責**：CLAUDE.md 管理、五個主 skill 管理、記憶管理、工作流程管理、四個流程的效率改善。確保一切為專案量身打造、無冗餘、無過度設計
- **不負責**：產品功能（→ 開發）、品質驗證（→ 審查）、系統保養（→ 維護）、問題分析與決策（→ 思考）
- **產出**：CLAUDE.md 調整、skill 增刪調整、記憶與工作流程維護、流程改善方案
- **從哪來**：思考的 commit 後反思結論、任何 skill 執行中發現的管理需求
- **往哪去**：直接修改 CLAUDE.md、skill 檔案、記憶和工作流程

---

## 核心原則

**優化的順序：消除 > 自動化 > 簡化 > 加速**

1. **消除**：這個步驟真的需要嗎？能不能完全去掉？
2. **自動化**：不能去掉的，能不能讓程式碼自動做？
3. **簡化**：不能自動化的，能不能減少步驟或降低難度？
4. **加速**：以上都不行，才考慮讓現有步驟更快

不要在錯誤的層次優化。加速一個不該存在的步驟是浪費。

## 零、Skill・記憶・工作流程・CLAUDE.md 管理

### 管理原則

- **為專案量身打造**：每條規則、每個章節都必須對應 OctoDock 的實際需求，不搬通用模板
- **無冗餘**：同一件事只在一個地方說。多處出現時，一處為源頭，其餘引用
- **無過度設計**：沒有實際觸發過的規則、沒有對應場景的流程，刪除而非保留
- **結構乾淨**：命名精準、層次分明、讀者 10 秒內能定位到需要的資訊

### Skill 管理

**增刪調整標準：**
- **增**：現有 skill 覆蓋不到的重複性多步驟流程
- **刪**：從來沒觸發的、被更高層 skill 涵蓋的、內容過薄可併入主 skill 的
- **調**：描述跟實際做法不一致的、觸發條件太模糊的
- **路由表同步**：skill 增刪後必須同步更新 CLAUDE.md 觸發索引

**結構品質檢查：**

| 檢查項 | 標準 |
|--------|------|
| Frontmatter | 每個 skill 必須有 `name` 和 `description`，description 足夠具體 |
| 職責邊界 | 五個主 skill 必須有「負責/不負責/從哪來/往哪去」 |
| 內容密度 | 過薄（< 10 行實質內容）的子 skill 應併入主 skill |
| 引用完整性 | skill 間的引用指向存在的檔案，路由表跟實際檔案一致 |
| 重複 | 同一張表、同一段描述不出現在兩個 skill 中 |

### 記憶管理

- **留下什麼**：從經驗提煉的認知、重複犯過的錯的通用教訓、用戶偏好、架構決策的「為什麼」
- **不留什麼**：能從程式碼讀出來的、一次性的任務細節、變化太快維護不了的
- **分層**：CLAUDE.md 是索引和路由 → Skills 是流程和規則 → Docs 是狀態和紀錄 → Code/DB 是實作本身
- **迭代**：同一個教訓出現兩次就寫入，規則太具體就抽象化，重疊的合併

### 工作流程管理

- 每次對話開始時讀 CLAUDE.md 確認路由表
- 對話過程中持續判斷當前工作是否有對應 skill 該載入
- 發現流程卡頓或重複時，評估是否需要新建或調整 skill

### CLAUDE.md 結構品質檢查

- 只負責路由和全局約定，不包含任何流程細節
- 路由表條目與 `.claude/skills/` 目錄完全對應
- 職責邊界表與各 skill 內的「職責邊界」段一致

## 一、優化開發流程

### 觀察指標

| 信號 | 說明 | 可能的瓶頸 |
|------|------|-----------|
| 同樣的設定步驟每次都要手動做 | 重複勞動 | 缺少自動化（腳手架、模板、hook） |
| 開發完才發現設計有問題 | 規劃不足 | 規劃階段跳太快 |
| Build 通過但實際行為不對 | 驗證不足 | 缺少端到端測試 |
| 改一處要同步改很多地方 | 耦合過高 | 架構需要重構 |
| 新增 App 要改的檔案太多 | 樣板程式碼多 | 核心層抽象不夠 |

### 優化方向

- **消除**：不需要的步驟直接刪。比如現有架構已經自動處理的，就不用手動做
- **自動化**：重複的檢查用 hook 自動跑（post-commit hook 已有前端同步提醒）
- **簡化**：新增 App 的最小工作量應該持續降低。現在 5 步，能不能 3 步？
- **加速**：build 時間、測試時間能不能縮短？

---

## 二、優化審查流程

### 觀察指標

| 信號 | 說明 | 可能的瓶頸 |
|------|------|-----------|
| 審查時才發現低級錯誤 | 品質問題應該更早被攔截 | 缺少自動檢查（lint、type check） |
| 每次審查都在看同樣的問題 | 審查清單沒沉澱 | 該加到 implementation-rules |
| 審查花太多時間在格式而非邏輯 | 格式問題佔比太高 | 需要更強的自動格式化 |
| 審查通過但上線後出問題 | 審查遺漏 | 審查維度不完整 |

### 優化方向

- **消除**：能被 lint/type check 自動抓的，不需要人工審查
- **自動化**：把反覆出現的審查項寫進 TypeScript interface（build 時強制）
- **簡化**：審查清單要精簡，20 項裡 5 項最關鍵的放最前面
- **加速**：審查的 5 個維度不是每次都要全跑，根據改動類型選擇性審查

## 三、優化維護流程

### 觀察指標

| 信號 | 說明 | 可能的瓶頸 |
|------|------|-----------|
| 總是忘記做維護 | 沒有提醒機制 | 需要排程或 hook |
| 維護時發現大量過期資料 | 自動清理沒跑或不完整 | 檢查清理排程 |
| 每次維護都在做同樣的事 | 維護步驟固定 | 應該自動化 |
| 維護後發現新問題比修的多 | 維護範圍太大 | 分批維護，每次聚焦一個面向 |

### 優化方向

- **消除**：資料保留政策執行得好，就不需要手動清理
- **自動化**：定期維護的檢查項能不能變成自動化腳本？
- **簡化**：把維護的 6 個面向按重要性排序，不是每次都全做
- **加速**：建立維護 dashboard，一眼看出哪裡需要關注

## 四、優化思考流程

### 觀察指標

| 信號 | 說明 | 可能的瓶頸 |
|------|------|-----------|
| 想太久才開始動手 | 過度分析 | 問題分類不夠快，該直接做的在猶豫 |
| 動手後才發現想錯方向 | 分析不到位 | 根因分析跳過了，直接跳到解法 |
| 同樣的認知重複提煉 | 認知沒沉澱 | 該寫進 CLAUDE.md 或 skill 的沒寫 |
| 認知寫了但從來沒觸發 | 觸發條件不對 | Skill 的 description 太模糊 |
| CLAUDE.md 越來越長但品質沒提升 | 知識膨脹 | 該合併、刪除、或提升抽象層次 |

### 優化方向

- **消除**：已知問題不需要每次重新思考，直接套用
- **自動化**：能用程式碼解決的認知（P0）比寫規則（P2）效果好 100 倍
- **簡化**：問題分類決策樹要直覺，3 秒內能判斷問題類型
- **加速**：建立模式庫（pattern library），常見問題的思考路徑直接複用

## 五、跨流程優化

### 流程間的接口

四個主流程不是獨立的，它們之間的接口也需要優化：

| 接口 | 常見問題 | 優化方向 |
|------|---------|---------|
| 開發 → 審查 | 開發完才審查，發現大問題要重做 | 規劃階段就做輕量審查 |
| 審查 → 維護 | 審查發現的技術債沒被追蹤 | 審查產出直接進維護清單 |
| 維護 → 思考 | 維護時發現模式但沒提煉 | 維護後固定跑一次認知進化 |
| 思考 → 開發 | 想出的改善沒被實作 | 認知進化的 P0（程式碼改動）要立即執行 |

### 迴路完整性

```
開發 → 審查 → 上線
  ↑                ↓
優化 ← 思考 ← 維護（觀察 production）
```

每個環節的產出都是下一個環節的輸入。如果某個接口斷了，整個迴路就失效：
- 開發沒審查 → 品質問題流到 production
- 審查沒維護 → 技術債累積
- 維護沒思考 → 同樣的問題反覆出現
- 思考沒優化 → 流程本身不會進步
- 優化沒回到開發 → 改善停在紙上

## 執行優化的流程

### 1. 觀察

- 哪裡花最多時間？
- 哪裡最常出錯？
- 哪裡每次都一樣？（= 可以自動化）
- 哪裡有重複？（= 可以消除或合併）

### 2. 找瓶頸

- 不是所有慢的地方都是瓶頸
- **瓶頸 = 改善後整體效率提升最大的那一個點**
- 同時改很多東西 = 什麼都沒改好

### 3. 對症下藥

- 用「消除 > 自動化 > 簡化 > 加速」的順序
- 一次改一個，驗證有效後再改下一個

### 4. 驗證

- 改完後，原本的問題信號消失了嗎？
- 有沒有引入新的問題？
- 改善是一次性的還是持續有效的？

### 5. 沉澱

- 有效的優化寫進對應的 skill
- 失敗的優化也記錄，避免重蹈覆轍

---

## 反模式

1. **過早優化** — 流程還沒穩定就開始優化，等於在流沙上蓋房子
2. **全面優化** — 同時改所有東西，無法判斷哪個有效
3. **優化體感而非瓶頸** — 改了你覺得慢的地方，但那不是真正的瓶頸
4. **只加不減** — 加了新步驟但沒刪舊步驟，流程越來越重
5. **優化無法衡量的東西** — 如果不知道現在的基線，就不知道改善了多少
